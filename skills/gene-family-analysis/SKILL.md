# 基因家族分析 Gene Family Analysis

本技能指导你(AI 助手)带领用户完成一个基因家族从“鉴定成员”到“讲出生物学故事”的完整分析。

## 通用原则 / Guiding principles

以下原则贯穿所有步骤,无需在每一步重复判断:

1. **可复现优先**:每一步都记录工具版本、数据库版本、关键参数,让结果能被他人
   (和审稿人)重做出来。
2. **终稿图表优先用脚本自定义**:正式发表用的图,优先用 **Python / R(ggplot2 / ggtree)等脚本自定义绘制**,以保证质量与辨识度, 避免 TBtools/GSDS 等工具默认输出造成的 “千篇一律”。
3.  GUI 工具可用于快速出草图或在无脚本权限时兜底,但**终稿建议脚本重绘**。
   (此原则的前提仍是“按当前可用权限选择”:无代码执行权限时退回 GUI 工具。)

## 何时使用 / When to apply

当用户要做下列任意一项时启动本流程:鉴定某物种某基因家族的全部成员、构建家族系统发育树、分析基因结构/保守基序/结构域、染色体定位、基因复制与共线性、Ka/Ks、启动子顺式元件、或表达模式分析。若用户只问其中一步,直接跳到对应小节,但仍提醒前置步骤(例如“做进化树前需要先确认成员清单的可靠性”)。

## 语言 / Language

默认使用**用户提问所用的语言**进行作答与输出;用户明确要求某种语言时,以其要求为准。
无论用中文还是英文输出,基因名、工具名、数据库名、统计量等专业术语一律**保留英文**
(如 Pfam、HMMER/hmmsearch、MAFFT、IQ-TREE、MEME、MCScanX、Ka/Ks、PlantCARE)。

Respond in the **language of the user's request** by default; honor any explicitly
requested language. Keep technical terms (gene/tool/database names, metrics) in English
regardless of output language.

## 开始前必须确认的输入 / Scoping

在动手前,先向用户确认以下信息;缺失项要主动追问,不要默认:

1. **物种与基因组版本**:物种学名、基因组来源与版本号(组装版本 + 注释版本)。
2. **家族定义依据**:家族名称,以及鉴定所依据的 **Pfam 结构域编号** 或 HMM 模型
   (例如 PF00010);若家族由多个结构域定义,需全部列出。
3. **用户已有的材料**:是否已有候选基因列表、蛋白序列、GFF、表达数据等。
4. **分析目标与终点**:是要发一篇基因家族鉴定文章、还是为后续功能研究筛选候选基因?
   终点不同,深度和取舍不同。

**基因组数据库与版本选择规则(重要,先确认再开工):**

- **优先使用该研究物种自己的专用基因组数据库**,而不是综合库。专用库的基因组与
  注释通常更新更及时、更贴近社区共识。例如:番茄 → SGN(https://solgenomics.net/);
  葫芦科作物 → Cucurbit Genomics Database(http://cucurbitgenomics.org/v2/)。
  更多物种的专用数据库见 `references/genome-databases.md`。
- **不优先使用 Phytozome / NCBI 作为序列来源**:它们收录的版本往往落后于物种专用库,
  可能不是最新组装/注释。可用于交叉核对,但以专用库为准。
- **存在多个组装版本时**,按以下优先级取最新、最完整的:
  T2T(端粒到端粒)完成图 > PacBio 等长读长(long-read)组装 > 最新发布版本。
- 选定后,**记录数据库名、组装版本号、注释版本号、访问日期**,写进 Methods 备查。

**从 NCBI 取基因组时的实战核对(踩坑提醒):**

- **认准 Assembly/Genome(GCA/GCF)记录,而不是 SRA**:有的提交者会把组装好的序列当作
  "reads"传进 SRA(SRX/SRR 名字里甚至写着 "genome assembly")。这种 SRA 记录不带标准
  注释,不能直接做家族鉴定——应改用对应的 **GCA 组装记录**(SRA 页 → BioProject →
  Assembly 即可找到)。
- **务必核实注释文件是否真的存在,别只信论文的 "Data availability"**:GenBank 的 GCA
  常只有基因组、无注释。下载前在该组装的 FTP 目录里确认是否有 `_genomic.gff.gz` 和
  `_protein.faa.gz`/`_cds_from_genomic.fna.gz`;只有 `_genomic.fna.gz` 则需另找注释。
  (论文宣称"注释见 Figshare/补充"却实际缺失的情况确有发生,必须亲自核对。)
  若该组装无注释,可改用同物种有 RefSeq 注释的版本(GCF_…),或自行重注释。
- **用物种浏览看全部可用组装**:`https://www.ncbi.nlm.nih.gov/datasets/genome/?taxon=<TAXID>`
  列出该物种所有组装,据此挑注释完整、版本最优者。**若同物种有多套种质组装**,则更适合
  做泛基因家族分析(改用 pan-gene-family-analysis 技能)。

## 工作流程总览 / Workflow

按以下顺序推进。每一步先说明“目的”,再给“标准做法”,最后留出你的经验补充。

### 1. 家族成员鉴定 Member identification

**目的**:得到一份可靠、无遗漏、无假阳性的家族成员清单。这是整个分析的地基,错了
后面全错。

**标准做法**:
- 用该家族的 Pfam 结构域 HMM 文件,对目标蛋白组做 `hmmsearch`(HMMER)。
- 同时用已知成员(常取拟南芥/水稻等模式物种)做 `BLASTp` 反向补漏。
- 合并两套结果、去重,得到候选集。
- **逐条验证结构域**:把候选蛋白提交到 NCBI-CDD / Pfam / SMART / InterPro,确认
  确实含有该家族的特征结构域;剔除不含核心结构域的假阳性。

**成员鉴定的关键参数与质控规则(作者经验):**

- **E-value 采用“先宽松、后过滤”策略**:初筛用较宽松的阈值(如 **0.01 / 1e-2**), 目的是宁可多收、不漏掉序列分歧较大的真成员(优先保证召回);随后通过**交叉比对**(HMM 与 BLAST 结果互验、结构域逐条核对)剔除假阳性。不要一上来就用极严阈值。
- **保留标准:核心关键结构域覆盖 > 30%**——候选蛋白必须含家族核心关键结构域,且该结构域的比对覆盖长度 **> 30%**(domain coverage > 30%);达不到的判为假阳性剔除。
- **可疑注释必须人工复核(两种情形都要查):**
  - *情形一*——**单个基因含不完整结构域**(如同一基因被注释出连续两个不完整结构域),提示该基因的 gene model 可能有误。
  - *情形二*——**相邻的两个基因各自只含一个不完整结构域**:它们很可能本是**同一个基因被错误拆分成了两个 gene model**。
  - 处理:回查原始注释的准确性,必要时用 **Fgenesh 等工具对相关基因座手动重新注释**;确认真实基因结构后,**再判断情形二的两个基因是否应合并为一个**,并据此决定最终取舍。

- **多转录本/多蛋白处理:只保留最长转录本**。当同一基因座存在多个转录本(isoform)时,统一只取**最长转录本(longest isoform / primary transcript)**进入下游分析,以保证每个基因在成员清单、建树、结构分析中只对应一条序列、避免重复计数。

### 2. 序列整理与命名 Sequence curation & naming

**目的**:为下游所有分析准备干净、对应关系明确的序列文件(基因组 ID、CDS、蛋白)。

**标准做法**:统一基因 ID;按染色体位置或系统发育关系给家族成员命名(如
*SpFAMILY1…n*);导出蛋白、CDS、上游启动子序列备用。

**命名规则(作者经验):**

总原则:**优先对齐拟南芥(Arabidopsis)同源基因所采用的家族命名规则**,让命名能与
已有文献相互对应。按以下逻辑确定:

- **物种前缀**:取拉丁学名首字母缩写(属名首字母 + 种加词首字母),如
  *Arabidopsis thaliana* → `At`。
  - 当**同时分析多个物种、且首字母缩写发生冲突**时:用「属名首字母 + 种加词的前 2 个字母」,仍冲突则取前 3 个字母,**逐步加长直到能区分各物种为止**。
- **家族成员编号**:
  - 若进化树**能区分亚家族(subfamily / clade)**:以亚家族命名,亚家族内**按染色体位置顺序**编号。
  - 若**不区分亚家族**:全部成员**按染色体位置顺序**统一编号。
  - 对**成员数较少的家族**:依据与拟南芥同源基因的**相似性比对**结果, **对应沿用拟南芥家族的编号**(使 `SpXXX` 与对应的 `AtXXX` 编号一致),便于跨物种对照。

**批量提取序列的工具(按当前权限选择):**

- 若**已获代码执行/脚本运行权限**:优先**下载基因组 fa + gff3 后自写脚本提取** (可复现、可批量,推荐)。
- 若**无脚本权限**:使用 **TBtools** 的序列提取功能完成。
- 即根据用户授予的权限自动选择上述其一,不要默认假设可以运行代码。

### 3. 理化性质 Physicochemical properties

**目的**:给出每个成员蛋白的基本参数,常作为文章第一张表。

**标准做法**:用 ExPASy ProtParam 批量计算**蛋白长度(Protein length, aa)**、分子量(MW)、理论等电点(pI)、不稳定指数、脂肪族指数、总平均亲水性(GRAVY)。表格按**家族编号顺序**列出,以上参数齐全即可,不需要额外补充其他指标。

**工具选择**:批量跑 ProtParam 推荐两种途径,按使用者情况选择——
- **TBtools**:操作更简单,适合学生/初学者,推荐优先教学生用这条路。
- **自建 Python 脚本**:适合需要更高自动化、可复现、可整合进流程的场景,同样推荐。
两者结果应一致,可任选其一,不强制要求用代码。

### 4. 亚细胞定位预测 Subcellular localization

**目的**:预测蛋白行使功能的细胞区室,为功能假设提供线索。

**标准做法**:用综合预测器(TargetP / WoLF PSORT / Plant-mPLoc / DeepLoc 等)交叉验证;
针对特定细胞器的问题改用专一工具(如 ChloroP、MitoProt、PredPlantPTS1);已知基因用
UniProtKB / SUBA4 交叉验证。

**完整工具清单(17+)、选用建议与命名对照见 `references/subcellular-localization.md`**
(已按用途分类整理:综合 / 细胞器专一 / 细菌专用 / 数据库)。先查阅该文件再做判断。

>该文件未覆盖的情形,可向用户说明需要进一步核实。


### 5. 系统发育分析 Phylogenetic analysis

**目的**:厘清家族成员间的进化关系、划分亚家族(subfamily/clade),是文章核心图之一。

**标准做法**:
- 多序列比对:MAFFT 或 ClustalW(蛋白序列)。
- 比对修剪:trimAl(去除 gap 过多的列),提高建树质量。
- 建树:IQ-TREE(先用 ModelFinder 选最优模型,最大似然 ML)或 MEGA(NJ/ML)。
- 支持度:Bootstrap ≥ 1000(或 IQ-TREE 的 ultrafast bootstrap / SH-aLRT)。
- 为划分亚家族,**纳入模式物种的已知成员**作为参照。

**建树方法与工具选择(作者经验):**

- **方法:ML 与 NJ 通常都做**,两种方法互相印证;结果一致则结论更稳。
  - **算力提醒**:当家族成员**数量 > 100** 且本机配置有限时,**ML(最大似然)会非常慢**。
    此时若环境**支持调用 MEGA**,可改用 **MEGA** 完成建树(NJ/ML)。
- **可视化/出图工具按数据情况选择:**
  - 若**有转录组数据、想把基因表达量与进化树结合展示**:推荐 **R / ggtree**
    (便于在树旁叠加表达热图等图层)。
  - 否则用常规美化工具(iTOL / MEGA 自带 / Evolview)即可。
- **参照物种(用于定义亚家族与命名,承接第 2 步命名规则):**
  - **双子叶植物(dicot)**:一般纳入**拟南芥 + 番茄**。
  - **单子叶植物(monocot)**:一般纳入**水稻**。
  - **求周全**:可同时纳入**拟南芥、番茄、水稻**三者一起建树。

> 待补充算法参数推荐

### 6. 基因结构 Gene structure (exon–intron)

**目的**:展示外显子—内含子结构,常与系统发育树并排呈现以显示进化关联。

**标准做法**:由 GFF3 注释文件绘制外显子—内含子结构。

**出图工具与美观度(作者经验):**
- **TBtools / GSDS 的 Gene Structure View 出图较丑、且高度同质化**——大量论文的结构图
  长得几乎一样,辨识度低。
- 为获得更美观、更有辨识度的图,**推荐改用 Python 脚本,或 R / ggplot2 自定义绘制**。
- 若**无脚本运行权限**,TBtools/GSDS 可用于快速出草图,但**终稿建议用脚本重绘**。


### 7. 保守基序 Conserved motifs

**目的**:识别家族保守的序列模块,辅助亚家族划分与功能推断。

**标准做法**:用 MEME Suite 预测保守 motif(需设定 motif 数量、宽度范围、位点分布等
参数),再可视化。出图遵循通用原则(终稿优先脚本自定义)。

**参数、解读与作图细节见 `references/motif-analysis.md`**(已据作者文章整理):
核心要点
——MEME 的阈值是 **motif 数量而非 E-value**,故须指定数量并逐个核查 E-value;
序列数>50 时优先用 STREME;
用 **Tomtom** 把 motif 注释/对应到已知 motif 与结构域。先查阅该文件再操作。

### 8. 保守结构域 Conserved domains

**目的**:在序列层面展示家族特征结构域分布。

**标准做法**:批量 CDD / InterProScan,TBtools 可视化结构域排布。

> 可与第 7 步合并出图。

### 9. 染色体定位 Chromosomal localization

**目的**:展示家族成员在染色体上的物理分布,为后续判断基因复制做铺垫。

**标准做法**:由 GFF 提取坐标,用 TBtools / MapChart / MG2C 作图。

**基因簇(gene cluster)判定规则(作者经验):**

基因簇是基因扩张的一种类型(常对应串联复制)。按以下标准判定:
- **物理距离窗口**:在染色体上设一个固定物理距离窗口,**窗口大小依物种平均基因密度
  选取,使其包含不少于 10 个基因**(常见如 **100 kb、200 kb**)。
- **判定条件**:若同一窗口内(或相邻)存在 **≥ 2 个功能相关的家族成员**,即判为同一
  基因簇。
- **空间分布 + 基因密度法**:通过家族成员之间**成对物理距离的空间分布**结合**全基因组
  基因密度**来识别基因簇,并**结合染色体定位图作可视化确认**。

### 10. 基因复制与共线性 Gene duplication & synteny

**目的**:解释家族扩张机制(串联复制 vs 片段复制 vs 全基因组复制),并通过种内/种间
共线性推断进化关系。这是把“鉴定”升级为“进化故事”的关键。

**标准做法**:
- MCScanX 鉴定共线性区块,区分 tandem / segmental / WGD 来源。
- 种间共线性(与拟南芥/近缘物种)作 Dual Synteny / 多物种共线性图(TBtools)。
- 选择压力:计算复制基因对的 **Ka/Ks**,判断纯化选择(<1)还是正选择(>1)。

**基因扩张的三种复制类型(判定要点,作者经验):**

植物基因家族规模变化通常来源于基因复制事件,主要分三类:
- **串联复制 Tandem duplication**:多个成员在染色体上**连续排列**(如 Chr3 上Gene1–Gene2–Gene3 相邻),提示由相邻基因复制产生;与上一步的"基因簇"高度对应。
- **片段复制 Segmental duplication**:成员位于**不同染色体**但相似性高;经验阈值**相似性 > 80%** 时,判为来自基因组片段复制(如 Chr1 的 GeneA 与 Chr5 的 GeneB)。
- **全基因组复制 WGD(Whole Genome Duplication)**:植物家族扩张的重要来源,表现为**基因数量整体增加**;许多代谢相关酶家族即源于 WGD。


### 11. 顺式作用元件 Cis-acting regulatory elements

**目的**:从启动子推测家族成员的潜在调控(激素、逆境、光、发育),为表达分析建立假设。

**标准做法**:提取起始密码子上游约 **2000 bp** 启动子序列,提交 PlantCARE;再把预测到
的顺式元件按功能归类、串联成支撑科学问题的"故事线",作统计/热图。

**归类方法见 `references/cis-element-classification.md`**(据作者经验整理):
核心流程——
先定故事线维度(胁迫/激素/生长发育等)→ 从 PlantCARE 的 **Description 列**提取关键词→ 映射到维度并合并同义元件 → 按基因/样本或按处理分组对比,用热图展示富集差异。
元件→维度的对照表也在该文件中,可持续扩充。先查阅该文件再操作。

### 12. 表达模式分析 Expression profiling

**目的**:用表达数据把候选基因和具体的组织/逆境/激素联系起来,筛选功能研究候选。

**标准做法与作者经验:**
- **数据来源**:优先到**本物种自己的基因组数据库**找公共转录组数据——这些数据库通常已收录该物种大量公共 RNA-seq(见 `references/genome-databases.md`,与第 1 步选库一致)。
- **分析目标**:检查家族成员在**不同处理或不同组织**中的**特异性表达**,据此筛选功能研究候选基因。
- **数据标准化**:常用 **FPKM / RPKM / TPM**——三者均**同时标准化样本总 read 数与基因测序深度**;其他还有 CPM、RPM 等。
- **作图**:用**热图 + 聚类**,推荐 **R / pheatmap**(`hclust` 默认方法聚类);出图遵循通用原则(终稿优先脚本)。热图前的数值变换**首选 Z-score**(log2 亦可)。
- **重点候选筛选标准**:在**不同处理或不同组织中表现出特异性表达**的家族成员,列为功能研究的**重点候选**;关键候选可再用 **qRT-PCR** 实验验证。

### 13.(可选)延伸分析 Optional extensions

**仅对前面筛出的少数重点候选基因加做**,用来深化"故事",不必对全家族铺开:

- **AlphaFold 三维结构预测**:解释家族成员之间的**细微差别**——如结合位点差异、结合力差异、蛋白活性差异等,把"序列层面的不同"落到"功能层面的不同",是讲清成员分化的有力证据。
- **miRNA 靶向预测**:从转录后调控层面**扩展家族成员的功能分析**,为功能假设增加新维度。
- **STRING 蛋白互作网络**:预测候选基因的互作伙伴,提示其参与的通路或复合体。

**关于 GO/KEGG 富集**:在基因家族成员分析中**用得很少、通常意义有限**——家族成员本就功能相近,对其自身做富集往往只是重复已知信息。
仅在分析**互作伙伴或共表达网络**等更大基因集时才考虑;对单个家族成员,用 GO **注释**(给每个基因贴 term)即可,不必做富集。
